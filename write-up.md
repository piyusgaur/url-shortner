# URL Shortener Take-Home Write-Up

I used AI as a pair-programming tool throughout this project, mainly to scaffold the repo structure, help break the work into clear milestones, and draft routine pieces like route shapes, test setup, and UI layout. I still made the product decisions myself: how duplicate URLs behave, how aliases are handled, what the data model should store, and how much analytics to expose. I also reviewed and corrected a few AI-generated ideas when they were more complicated than the assignment needed, especially around frontend configuration and the amount of UI surface area.

The biggest implementation choice was to make shortening idempotent: if the same long URL is shortened twice, the service returns the existing short code instead of creating a second one. That kept the behavior easy to explain and avoided confusing duplicate mappings. I also chose to treat custom aliases as explicit codes with validation, so they remain URL-safe and predictable. For short-code generation, I used cryptographically random URL-safe strings with uniqueness checks in the datastore rather than a sequential counter, because it keeps the codes compact, avoids guessable patterns, and is straightforward to reason about in a take-home setting.

Another tradeoff was keeping the analytics intentionally lightweight. The redirect flow already tracks click count and last accessed time, so I exposed a simple analytics endpoint and a small frontend panel instead of building charts, dashboards, or historical reporting. That keeps the project focused on the core assignment while still demonstrating end-to-end thinking. On the frontend, I spent effort on clarity and responsiveness rather than adding more features: the main flow is shorten, review the result, and inspect usage from the same screen.

What I overrode or threw away was mostly complexity. I removed heavier layout ideas, kept the config simple when Vite and Windows/OneDrive path resolution got in the way, and avoided adding dependencies that would not make the submission stronger. The final shape is intentionally small, readable, and easy to walk through in a follow-up conversation.

What is still missing is the kind of production hardening I would add with more time: authentication and ownership for analytics, richer reporting over time, integration tests against a real MongoDB instance, deployment-specific environment setup, and a little more polish on copy-to-clipboard and sharing flows. Even without those extras, the repository shows the full shorten -> redirect flow, documented tradeoffs, and a clean commit history that reflects how the project came together.
